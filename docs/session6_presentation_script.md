## 2. 시나리오별 설명 포인트

### 시나리오 A: 정상 부하(기준선)
- 왜 필요한가:
  - 피크나 장애 실험은 기준선이 있어야 해석 가능하다.
  - 정상 상태의 처리시간과 완전성을 먼저 확보해야 한다.
- 무엇을 봤는가:
  - Producer 완료 시간
  - 첫 DB 반영 시간
  - 마지막 DB 반영 시간(E2E)
  - row completeness
- 어떻게 해석하는가:
  - 기준선 자체가 불안정하면, 이후 문제는 설계가 아니라 기초 자원 부족일 수 있다.

### 시나리오 B: 버스트 트래픽
- 왜 현실적인가:
  - 생산라인 재가동, 네트워크 복구, 교대 시작 시 이벤트가 짧게 몰린다.
- 무엇을 봤는가:
  - lag peak
  - backlog drain time
  - 최종 누락 여부
- 어떻게 해석하는가:
  - lag는 올라갈 수 있지만, 일정 시간 내 회복하고 누락이 없으면 운영 가능 범위다.

### 시나리오 C: 채널별 추론 시간 차이
- 왜 필요한가:
  - 지금은 정상 판정 고정이지만, 실제 운영에서는 모델 추론 시간이 지연을 만든다.
  - `laser_a`, `laser_b` 모델이 서로 다르면 채널별 병목이 다르게 나타난다.
- 무엇을 봤는가:
  - 채널별 ingest->DB latency
  - 채널별 처리량 차이
- 어떻게 해석하는가:
  - 느린 채널이 전체 SLA를 끌어내리는지 확인하고, 컨슈머 배치/리소스 조정 근거로 사용한다.

### 시나리오 D: Spark consumer 장애 복구
- 왜 현실적인가:
  - 로컬/운영 모두 프로세스 중단은 반드시 발생한다.
  - 중요한 것은 “안 멈춤”이 아니라 “멈춰도 복구되고 누락이 없는가”다.
- 무엇을 봤는가:
  - MTTD(감지 시간), MTTR(복구 시간)
  - 복구 후 backlog 소진 시간
  - 누락/중복 여부
- 어떻게 해석하는가:
  - 감지 및 복구 자동화가 되면 수동 운영 부담이 크게 줄어든다.

## 3. Q&A 대비 핵심 답변

### Q1. 왜 버스트 테스트가 중요한가?
- “평균 처리량만으로는 실제 공장 이벤트 집중 구간을 설명할 수 없어서, 순간 집중 트래픽 복원력을 확인하기 위해서입니다.”

### Q2. 모델이 정상만 판단하는데 왜 추론시간을 넣었나?
- “정상/비정상 판정값과 별개로, 추론 연산 시간 자체가 큐 적체를 만들 수 있기 때문에 지연 모델링은 필수입니다.”

### Q3. 장애 시 왜 Kafka를 두는가?
- “컨슈머가 죽어도 broker에 데이터가 남아 복구 후 이어서 처리할 수 있기 때문입니다.”
