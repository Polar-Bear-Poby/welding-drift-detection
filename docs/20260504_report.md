## 1. 서비스 이용자 효용

**이 서비스는 용접 불량 배터리가 다음 공정으로 넘어가기 전에 자동으로 차단한다.**

용접 공정이 끝나면 레이저 신호(laser_a, laser_b)를 실시간으로 분석한다. 분석 결과는 즉시 데이터베이스에 기록되고, 대시보드에서 바로 확인할 수 있다.

이용자가 얻는 효용은 세 가지다.

**① 불량 조기 차단**  
신호 드리프트가 탐지되면 `drift`로 판정된다. 해당 배터리가 어떤 라인에서, 몇 번 제품에서 발생했는지 즉시 추적할 수 있다. 불량이 출하까지 흘러가는 것을 막는다.

**② 생산라인별 현황 실시간 확인**  
대시보드(Streamlit)에서 라인별·채널별 정상/드리프트 건수를 실시간으로 볼 수 있다. 어떤 라인에 문제가 집중되는지 한눈에 파악된다.

**③ 장기 품질 추세 자동 집계**  
Airflow가 매일/매주 자동으로 리포트를 만든다. 추세가 나빠지면 경보가 발생한다. 예방 정비 시점을 데이터로 판단할 수 있다.

## 2. 카프카·스파크·에어플로우의 역할

**Kafka는 생산라인과 분석 시스템 사이의 실시간 버퍼다. Spark는 신호를 분석해 품질을 판정한다. Airflow는 배치 파이프라인 전체를 자동으로 실행한다.**

### Kafka
생산라인 N대가 동시에 데이터를 전송해도 Kafka가 모두 받아 저장한다. 소비자(Consumer)가 느리거나 일시 장애가 생겨도 데이터를 잃지 않는다. offset을 되감으면 언제든 다시 처리할 수 있다.

채널별 토픽(`welding.raw.laser_a.v1`, `welding.raw.laser_b.v1`)을 분리했다. 덕분에 laser_a와 laser_b를 독립적으로 처리하고 확장할 수 있다.

### Spark
Kafka에서 읽은 청크 신호를 배터리 1개 단위로 재조립한다. 16구간으로 분할해 홀수/짝수 평균을 비교하고 cpd_score를 계산한다. score가 임계값 이상이면 `drift`, 미만이면 `normal`로 판정한다.

판정 결과는 PostgreSQL(`welding.pattern_summary`)에 저장된다. Airflow가 여기서 집계해 일간/주간 리포트를 만든다.

### Airflow
9개 DAG가 파이프라인 전체를 자동으로 관리한다.

| DAG | 역할 |
|---|---|
| `welding_batch_ingest` | 라인별 프로듀서를 병렬 실행하고 Spark 배치를 구동 |
| `welding_batch_backfill` | 과거 날짜 데이터를 수동 재처리 |
| `welding_daily_quality_report` | 매일 라인·채널별 정상/드리프트 건수를 집계 |
| `welding_weekly_drift_trend` | 주간 cpd_score 추세와 장기 드리프트 여부를 판정 |
| `welding_streaming_monitor` | Spark Consumer가 정상 동작 중인지 헬스체크 |
| `welding_consumer_health_monitor` | Kafka consumer group 멤버 수를 검증 |
| `welding_kafka_topic_health` | 토픽 lag이 임계값을 초과하면 경보 |
| `welding_data_availability_check` | pattern_summary 행 수로 데이터 수집 여부 확인 |
| `welding_storage_cleanup` | Parquet, 로그, Spark 체크포인트 정기 정리 |

## 3. 파이프라인 구성도 업데이트 (7차시)

> 상세 Mermaid 구성도 및 차시별 이력은 [`welding_drift_pipeline_updated.md`](welding_drift_pipeline_updated.md) 참조.

7차시에서 파이프라인에 적용된 핵심 변경 세 가지:

### ① 1 Line = 1 Producer 병렬화 (논리 결함 수정)
`welding_batch_ingest` DAG가 기존에는 프로듀서 컨테이너 1개에 `--line-count N`을 전달해 N개 라인을 **순차 전송**했다. Airflow Dynamic Task Mapping으로 교체해 라인별 Task를 **병렬** 실행하도록 수정했다.

```
이전: [producer 1개] → LINE_01 전송 완료 → LINE_02 전송 완료 → LINE_03 전송 완료 (순차)
수정: [Task-1 LINE_01]  ─┐
      [Task-2 LINE_02]  ─┼→ Kafka (동시)
      [Task-3 LINE_03]  ─┘
```

### ② quality_decision 값 체계 전환
배치와 스트리밍 모두 `"normal"` / `"drift"` 로 통일했다. 기존 `"PASS"` 값은 legacy로 유지하고 SQL에서 `IN('PASS', 'normal')`로 병행 처리한다.

### ③ API + 대시보드 추가
FastAPI와 Streamlit을 새로 추가했다. DB 적재 결과를 조회하고, 룰 기반 추론 결과를 시각화할 수 있다.

## 5. 논리 결함 발견 및 해결 — 장애 대응 실험 재개

**생산라인 N대가 병렬로 데이터를 전송해야 하는 설계를, 실제로는 1개 프로듀서가 순차적으로 처리하고 있었다. 오늘 이 결함을 코드에서 확인하고 수정했다.**

### 발견한 결함

`welding_batch_ingest` DAG의 기존 코드:

```python
# 결함: 프로듀서 1개가 --line-count 3으로 3개 라인을 순차 전송
run_producer = BashOperator(
    bash_command=(
        "docker exec welding-producer python /app/producer.py "
        "--line-count 3 ..."  # LINE_01 → LINE_02 → LINE_03 순서대로
    )
)
```

`producer.py` 내부의 `make_publish_items()` 함수는 `line_count=3`을 받으면 제품 목록을 라인별로 순환(round-robin)하며 **단일 스레드로 순차** 발행했다. 생산라인 3대가 **동시에** 전송하는 것처럼 보이지 않았다.

### 수정 방법

Airflow Dynamic Task Mapping으로 라인별 Task를 병렬 생성했다.

```python
@task()
def run_producer_for_line(line_number: int):
    cmd = (
        f"docker exec welding-producer python /app/producer.py "
        f"--line-number {line_number} ..."  # 라인 1개만 담당
    )
    subprocess.run(cmd, ...)

line_numbers = list(range(1, REPLAY_LINE_COUNT + 1))
run_producer_for_line.expand(line_number=line_numbers)
```

`producer.py`에는 이미 `--line-number` 인자가 구현되어 있었다. Task N개가 동시에 실행되어 각각 `LINE_01`, `LINE_02`, ... 을 독립적으로 전송한다. `producer_count == line_count` 불변식이 DAG 레벨에서 보장된다.

### 향후 실험 계획

논리 결함 수정으로 진정한 병렬 전송 환경이 갖춰졌다. 계획했던 장애 대응 실험을 이제 올바른 조건에서 수행할 수 있다.

- **시나리오**: 생산라인 3대 병렬 전송 중 라인 1대(Task)가 실패했을 때 나머지 2대가 정상 처리되는지 검증
- **검증 기준**: `pattern_summary`에 LINE_02, LINE_03 데이터는 정상 적재되고, LINE_01은 재처리 대상으로 기록되는지 확인
- **복구 검증**: `welding_batch_backfill` DAG로 LINE_01을 재처리했을 때 데이터 연속성이 유지되는지 확인
