# 배터리 용접 드리프트 탐지 파이프라인 - 4차시 진행 상황 보고

본 문서는 4차시 개발 내용 및 방금 실시한 엔드 투 엔드(End-to-End) 파이프라인 실험 결과를 요약한 보고서입니다.

## 1. 4차시 주요 개발 성과

### 🛠️ 데이터 수집 및 시뮬레이션 환경 고도화
- **Kafka Producer 리플레이 엔진 구현**: 실시간 센서 API 없이도 각 생산 라인별로 생성되는 CSV 파일을 읽어 Kafka로 스트리밍하는 리플레이 엔진을 구축했습니다.
- **결정론적 리플레이**: 랜덤 시드(Seed)를 활용해 여러 생산 라인의 데이터 발생 순서를 재현할 수 있도록 기능을 추가했습니다.

### 📊 Spark 배치 처리 및 분석 로직 최적화
- **신호 패턴 분할**: 수집된 원시 용접 신호를 16개의 균등한 패턴으로 분할하여 분석하는 로직을 Spark Batch로 구현했습니다.
- **드리프트 탐지 지표(CPD Score)**: 홀수/짝수 패턴 간의 평균 편차를 활용한 드리프트 탐지 대리 지표(`cpd_score`)를 산출하고 품질 판정(`quality_decision`)을 수행합니다.

### 📈 성능 메트릭 및 로깅 시스템 도입
- **처리 속도 측정**: Kafka 및 Spark 스크립트 전반에 초당 처리량(`msg_per_sec`, `files_per_sec`) 및 소요 시간을 측정하는 로직을 추가했습니다.
- **영구 로그 저장**: `storage/logs/` 폴더 내에 `producer.log`와 `spark_batch.log`를 생성하여 성능 지표를 추적할 수 있도록 했습니다.

## 2. 파이프라인 통합 실험 결과 (2026-04-23)

### 🧪 실험 시나리오
- **대상**: 30개 제품(Product) 데이터 시뮬레이션 및 분석
- **흐름**: `Producer (Kafka 발행)` -> `Kafka Broker` -> `Spark Batch (분석/가공)` -> `PostgreSQL (결과 적재)`

### 📊 실험 결과 요약
- **Kafka 발행**: 총 **7,987개**의 메시지 청크를 안정적으로 발행했습니다. (평균 속도: ~26 msg/sec)
- **Spark 처리**: 60개의 파일을 분석하여 총 **944개**의 세그먼트 데이터를 생성했습니다.
- **DB 적재**: 분석된 품질 판정 데이터가 `welding.pattern_summary` 테이블에 성공적으로 저장되었습니다.

#### [성능 지표]
- **Spark 처리 총 소요 시간**: 약 121.67초
- **초당 처리 파일 수**: 0.49 files/sec
- **데이터 판정 결과**: 30개 제품 모두 **PASS** 판정 (드리프트 탐지 스코어 정상 범위 내)
